# Proxmox templates for Windows Server

Packer automation that builds cloud-init-ready **Windows Server 2022 Desktop** and **Windows Server 2025 Desktop** templates on **Proxmox VE 9** using **Cloudbase-Init**.

Cloned VMs get hostname, admin user password, SSH public keys, network config and more from Proxmox's cloud-init drive automatically.


## Prerequisites

- Proxmox VE 9 host with API reachable.
- Proxmox API token for an automation user (like `packer@pve!packer`) with sufficient permissions.
- Uploaded ISO images to Proxmox datastore:
  - Windows Server 2022 ISO
  - Windows Server 2025 ISO
  - VirtIO drivers ISO ≥ `virtio-win-0.1.240`
- Local Linux machine (or WSL on Windows) with [pixi](https://pixi.sh) installed. No need to install Packer, pixi will take care of it, in an isolated environment.


## Quick start

```bash
pixi install

# first time per variant
cp pkrvars/2025-desktop.pkrvars.hcl.example pkrvars/2025-desktop.pkrvars.hcl
# set proxmox_url, node, storage, username, network bridge, ISOs, template parameters
$EDITOR pkrvars/2025-desktop.pkrvars.hcl

export PKR_VAR_proxmox_api_token='<token-secret>'

./build.sh 2025-desktop  # or: pixi run build-2025
./build.sh 2022-desktop  # or: pixi run build-2022
```

This produces a Proxmox template with `vmid` and `vm_name` specified in `pkrvars/*.pkrvars.hcl`.


## Features of VMs cloned from the templates

- VM with UEFI firmware, SecureBoot and GPT partitioning scheme.
- Each clone gets unique `SID` and `Generation ID`, because the template is generalized by `sysprep`.
- Cloud-init:
  - When `CloudConfig Drive` hardware component is present and **its content is generated by Proxmox**, Cloudbase-Init applies those settings from it:
    - Hostname set to VM name.
    - Admin user (= `Administrator` for now) password.
    - Admin user SSH public keys. **Require ONE MORE manual reboot to apply correctly.**
    - Network config (IP address, gateway, DNS server, search domain).
    - Windows Update setting (enabled/disabled).
  - Changing admin username via `--ciuser` doesn't work out of the box, because Cloudbase-Init expects it in a different place than Proxmox uses. It can be done with custom cloud-init content, but we do not cover it here.
  - There are more settings that can be applied, but that requires custom cloud-init content (ISO images, file snippets, etc.) set via `--cicustom` or manual attachment of the ISO as a CDROM drive, which we do not cover here.
  - When you don't specify admin user password via `--cipassword` or there is no `CloudConfig Drive`, Cloudbase-Init sets it to a random password and enables the requirement to change it on first login. You can then setup the password using VM console in Proxmox UI.
  - If you change cloud-init settings and regenerate cloud-init content in Proxmox, it will be applied after the next reboot. You can use this to recover from lost admin user password scenario, for example.

- QEMU Guest Agent installed and enabled.
- WinRM enabled and configured with HTTPS listener and self-signed certificate with CN matching the hostname. Allowed in Windows Firewall for **all profiles**. Works with Ansible out of the box.
- RDP enabled with required NLA. Allowed in Windows Firewall for **all profiles**.
- **WINDOWS SERVER 2025 ONLY:** SSH server enabled with a part of the default configuration related to special `C:\ProgramData\ssh\administrator_authorized_keys` file **disabled**, see comments in `01-enable-ssh.ps1`. Allowed in Windows Firewall for **all profiles**.


## Clone flow

```bash
TEMPLATE_VMID=9025
NEW_VMID=210

qm clone $TEMPLATE_VMID $NEW_VMID --name win2025-app01 --full
# not all settings are demonstrated
qm set $NEW_VMID --cipassword 'S3cure!Pass'
qm set $NEW_VMID --ipconfig0 'ip=10.0.0.50/24,gw=10.0.0.1' --nameserver 10.0.0.1
# regenerate cloud-init content
qm cloudinit update $NEW_VMID
qm start $NEW_VMID
```

Cloudbase-Init applies settings ~2–3 min after first boot.

**It's important to remove the `CloudInit Drive` hardware component (via UI or `qm`) after the first boot, otherwise it can cause issues with subsequent boots (re-applying some settings).**


# Licensing

Use following commands to manage Windows activation status and product key:

```powershell
# Check activation status, product key, etc.
slmgr.vbs /dlv

# Uninstall product key and deactivate Windows.
slmgr.vbs /upk
```

Modern Windows Server ISO images (2022, 2025) contain multiple editions, and the edition is selected during template creation by Packer (see `pkrvars/*.pkrvars.hcl` variable `vm_os_edition`).

The product key given when activating the cloned VM will automatically switch variant to Retail, MAK or KMS, depending on the type of the key, so you can use the same template for all three activation types.

For evaluation ISOs, you need to convert the cloned VM from Eval to Retail, and at the same time activate it with either Retail, MAK or KMS key:

```powershell
dism.exe /Online /Set-Edition:ServerDatacenter /AcceptEula /ProductKey:<key>
# or
dism.exe /Online /Set-Edition:ServerStandard /AcceptEula /ProductKey:<key>
```
